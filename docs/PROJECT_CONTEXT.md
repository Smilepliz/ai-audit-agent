# Контекст проекта — SiteAuditBot

## 1. Что это за проект

**SiteAuditBot** — инструмент командной строки для автоматического коммерческого аудита сайтов. Анализирует одну страницу сайта (обычно главную), собирает SEO-метаданные, проверяет конверсионные элементы, сигналы доверия, ecommerce-функциональность и технические аспекты. Результат — Markdown-отчёт с оценками и рекомендациями.

Текущая версия: **v0.5**. Кодовая база — Python-пакет `audit/`, работающий через `python -m audit <url>`.

## 2. Для кого он делается

- **Владельцы небольших и средних сайтов** — получение быстрой диагностики без найма SEO-специалиста.
- **SEO-специалисты и веб-мастера** — автоматизация первичного аудита, массовая проверка клиентских сайтов.
- **Разработчики** — интеграция в CI/CD пайплайны для регрессионного контроля качества сайтов.

## 3. Какую проблему решает

Ручной аудит сайта — трудоёмкая и дорогая процедура. Существующие онлайн-сервисы либо платные, либо ограничены по глубине анализа, либо не адаптированы под российский рынок (требования к языку отчёта, специфика SEO-проверок). SiteAuditBot предоставляет **бесплатную, быструю, полностью автоматическую** первичную диагностику сайта с упором на коммерческие факторы (конверсия, доверие, ecommerce).

## 4. Что уже реализовано

### Базовая инфраструктура
- CLI через `argparse` с поддержкой опционального `--config`
- HTTP-загрузка страницы через `requests`, обработка таймаутов и ошибок
- Антибот-детекция (маркеры Cloudflare, DDoS-Guard, JS-shell)
- Оценка пригодности ответа (`assess_input_quality`)
- Сохранение отчётов в `reports/<domain>/latest.md` + архив по дате

### SEO-проверки
- Title (длина, качество, ключевые слова)
- Description (длина, качество, дублирование с title)
- H1 (наличие, качество, дублирование с title)
- Canonical (наличие, самоссылочность)
- Hreflang (наличие, корректность)
- Schema.org / JSON-LD (наличие, типы)
- Alt-атрибуты изображений (статистика отсутствующих и пустых)
- robots.txt (наличие, запрещённые разделы)
- Sitemap.xml (наличие, доступность)

### Конверсионные проверки
- CTA-элементы (сильные и слабые призывы к действию)
- Контактная информация (телефон, email, адрес, мессенджеры)
- Сигналы доверия (сертификаты, лицензии, отзывы, гарантии, партнёры)

### Технические проверки
- HTTP-заголовки (X-Robots-Tag, security-заголовки)

### Ecommerce
- Классификация сайта (4 типа + unknown) на основе эвристик
- 16 пунктов ecommerce-чеклиста (каталог, корзина, цены, карточки товаров, кнопки покупки)
- Ecommerce-проверки с категоризацией проблем

### Архитектура
- **v0.5**: Монолит `audit.py` (~2214 строк) разбит на модульный пакет `audit/` (16 модулей)
- Модели данных через `TypedDict`
- `AuditConfig` с возможностью загрузки из JSON/YAML
- Чёткое разделение: загрузка → парсинг → проверки → классификация → отчёт

## 6. Важные архитектурные ограничения

### Неизменяемые решения (не менять без отдельного обсуждения)
- **Одно-страничный аудит** — проверяется только главная страница (один HTTP-запрос). Многостраничный обход — в планах (v0.7).
- **Синхронный HTTP** — используется `requests`. Асинхронность — в планах (v1.0).
- **Только статический HTML** — без JS-рендеринга. SPA-сайты детектируются, но не анализируются. Playwright — в планах (v1.0).
- **Формат моделей** — `TypedDict`, а не `dataclass` (решение из modular-architecture-review).
- **Язык отчёта** — русский.
- **Формат вывода** — только Markdown (JSON и HTML — в планах).
- **Отсутствие тестов** — тесты планируются в v0.7, до этого рефакторинг только со сравнением отчётов.

### Текущие технические ограничения
- Нет CI/CD, нет линтера, нет type checking в CI
- Нет хранения истории аудитов (SQLite — post-1.0)
- Нет плагинной системы (проверки жёстко зашиты в модули)

## 7. Как правильно развивать проект дальше

### Принципы
1. **Модульность** — каждое новое расширение должно добавляться как отдельный модуль, не затрагивая существующие.
2. **Неразрушающий подход** — новые функции (например, DeepSeek) работают через опциональные флаги и не меняют поведение без них.
3. **Сохранение обратной совместимости** — CLI-интерфейс, формат отчёта, публичные функции не ломаются.
4. **Сначала план, потом код** — перед реализацией составляется план, который утверждается (как в INSTRUCTIONS.md).
5. **Минимально инвазивные изменения** — предпочтение небольшим, целенаправленным правкам перед масштабным рефакторингом.
6. **Сравнение отчётов до/после** — при любых изменениях, влияющих на логику проверок, отчёты должны быть идентичны.

### Порядок действий для новой фичи
1. Изучить существующие планы в `plans/` (нет ли уже проработанного подхода).
2. Составить план с разделами: цель, какие файлы создаются/изменяются/не трогаются, риски, порядок реализации, проверки.
3. Получить подтверждение плана.
4. Реализовать строго по плану.
5. Проверить регрессию (сравнение отчётов, запуск без новых флагов).
6. Обновить PROJECT_CONTEXT.md если структура проекта изменилась.

### Кого уведомлять об изменениях
- Изменения в архитектуре, CLI, формате отчёта — требуют обсуждения.
- Добавление новых проверок в существующие категории — safe, план не обязателен.
- Добавление новых зависимостей — требуется отметить в плане.
